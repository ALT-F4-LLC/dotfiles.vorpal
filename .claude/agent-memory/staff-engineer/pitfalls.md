# Recurring Pitfalls — staff-engineer

- Historical-audit signal re-fires for an already-encoded lesson → cross-project pitfalls memory is never cleared, so the same fem-kubernetes render-mandate signal surfaced in 3 consecutive evolve-skills cycles (2026-06-04 add, 2026-06-08 no-op, 2026-06-09 no-op) against design-qa → before drafting any CHANGE from an audit signal, grep the target skill AND its changelog for the lesson first; a verified already-encoded signal is a no-op verdict, not a re-add.
- evolve-* historical audits report FP-dominated operator-correction counts (3 consecutive cycles, 2026-06-09 cycle 6/6 raw hits false) → the correction-phrase regex (`didn't work|actually|broken|...`) matches audit-report prose echoed inside teammate-message relays and command output replicated into user-role transcript turns, not operator typing → filter to operator-typed turns (exclude turns containing `<teammate-message`/`<command-name>`/`tool_result` markers) before counting; until the skill encodes this, treat raw correction counts as ~0 true positives.
- Bash snippets in skill bodies silently corrupted by argument substitution → Claude Code expands unescaped `$`+digit in SKILL.md content as 0-based positional args (doc-confirmed v2.1.163+: `$N` = `$ARGUMENTS[N]`, escape is `\$N`), so a shell-function recipe using `$1`/`$2`/`$3` gets its params replaced by stray invocation words → when auditing/authoring any skill containing bash recipes, grep `grep -nE '\$[0-9]'` and escape hits as `\$N`; substitution restores the literal `$N` before the agent reads the body (found in review-and-comment 2026-06-09, the only affected skill of 15 that cycle).
- Pre-flight hard-refusal gate ordered AFTER an interactive overwrite dialog → numbered procedure steps drift from the Failure Mode table's precedence qualifiers (e.g., prd's "file exists (and slug is not reserved)" implied reserved-check-first, but steps ran collision dialog at 4 and refusal at 5, exposing "Overwrite" on init-specs-owned reserved specs) → when reviewing any skill with both a refusal gate and a collision/overwrite dialog, verify the abort-class check precedes the interactive dialog in step order, not just in the table (found in prd 2026-06-09).
- Skill bodies instruct direct `Skill(vote, ...)` while the calling agent's spec bans it in team mode → report-emission skills' Save & Return / When-NOT-to-Use sections predate (or drifted from) the vote Delegation Protocol, so they tell the caller to nest a team (banned by every agent spec's CRITICAL banner; correct path is `docket vote create` + delegation_request to team-lead) → when reviewing any skill that mentions vote escalation, grep `Skill(vote` and require mode-split phrasing (standalone: direct; team: delegation); found in design-review:237/85, code-review:390/96, verify-ac:261/93 (2026-06-09 cycle — design-review fixed, siblings routed to Phase 2).
- Prior-cycle "verified accurate" verdicts on cross-file references treated as still valid → a changelog no-change entry records verification against a SNAPSHOT (referenced agent file, CLI version), and both drift — simplify-scout's 12-principle table was verified 2026-06-08 but agents/senior-engineer.md carried uncommitted evolve-agents modifications by 2026-06-09, and the 2026-05-29 changelog's "bundled /simplify renamed → /code-review in 2.1.147" claim was false against the live v2.1.170 skill list (bundled /simplify exists, applies fixes) → before reusing any prior-cycle verification, check `git status` for modifications to the referenced file and re-confirm CLI/feature claims against the current environment; re-verify rather than cite the old changelog.
