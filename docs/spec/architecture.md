# Architecture Specification

## Purpose

This repository is a declarative dotfiles system implemented as a Rust crate on top of Vorpal. The repository source defines development tooling, user environment artifacts, generated configuration files, and AI-agent/skill assets that are published into host dotfile locations through Vorpal-built artifacts and symlinks.

The observed build has two top-level artifacts:

- `dev`: the development toolchain used to build the project, currently Protoc and a Rust toolchain.
- `user`: the user environment, including CLI tools, generated configs, scripts, copied agent trees, copied skill trees, environment variables, and host symlinks.

## Entry Points

The executable entry point is the `vorpal` binary declared in `Cargo.toml` at `src/vorpal.rs`.

`src/vorpal.rs` performs the top-level build orchestration:

- obtains a Vorpal `ConfigContext` with `get_context().await`;
- builds Protoc and the Rust toolchain;
- derives Rust toolchain environment variables for the `dev` artifact;
- builds `DevelopmentEnvironment::new("dev", SYSTEMS.to_vec())`;
- builds `UserEnvironment::new("user", SYSTEMS.to_vec())`;
- calls `context.run().await` after artifact declarations are registered.

`Vorpal.toml` identifies the project language as Rust and limits Vorpal source inputs to `src`, `Cargo.toml`, and `Cargo.lock`. Other repository directories such as `agents`, `skills`, `docs`, and `.github` are not part of the declared Vorpal source include list, even though `src/user.rs` references `agents/*` and `skills/*` through `FileSource`.

## Platform Model

The library exports `SYSTEMS` from `src/lib.rs` with four Vorpal artifact systems:

- `Aarch64Darwin`
- `Aarch64Linux`
- `X8664Darwin`
- `X8664Linux`

The repository README states macOS on Apple Silicon is the primary supported platform. The checked-in `Vorpal.lock` currently records source entries for `aarch64-darwin`, so lockfile coverage observed during this pass is narrower than the `SYSTEMS` constant.

## Module Boundaries

The Rust crate has three top-level source modules:

- `src/vorpal.rs`: binary entry point and artifact orchestration.
- `src/lib.rs`: shared exports for supported systems and Vorpal store output path construction.
- `src/file.rs`: reusable file artifact helpers.
- `src/user.rs`: user environment assembly and integration of tools, generated configs, copied sources, environment variables, and symlink targets.

`src/file.rs` defines the low-level file artifact primitives:

- `FileCreate`: creates a file from in-repo string content, optionally executable, using a shell step and `chmod`.
- `FileDownload`: downloads or sources a remote path into a Vorpal artifact using `ArtifactSource`.
- `FileSource`: copies an in-repo path into a Vorpal artifact using `ArtifactSource::new(..., ".").with_includes(...)`.

`src/user.rs` owns the high-level user environment composition. It imports typed configuration generators from `src/user/*` and artifact definitions from `vorpal-artifacts` and `vorpal-sdk`. The module builds third-party CLI artifacts, generated config artifacts, script artifacts, copied agent and skill trees, then wires them into `artifact::UserEnvironment`.

The `src/user/*` modules are typed configuration generators:

- `bat.rs`: builds the Bat config content.
- `claude_code.rs`: serializes Claude Code settings to pretty JSON through Serde.
- `codex.rs`: serializes Codex configuration to pretty TOML through Serde and `toml`.
- `ghostty.rs`: builds Ghostty key-value config content.
- `k9s.rs`: builds K9s skin YAML content.
- `opencode.rs`: serializes OpenCode settings to pretty JSON through Serde.
- `statusline.sh` and `teammate-idle-hook.sh`: shell scripts embedded through `include_str!` and emitted as executable file artifacts.

## Artifact Graph

The `dev` artifact depends on:

- `Protoc::new().build(context)`
- `RustToolchain::new().build(context)`

The `user` artifact builds CLI tool artifacts from `vorpal-artifacts` and `vorpal-sdk`, including AWS CLI, Bat, Direnv, Doppler, fd, fzf, GitHub CLI, Git, gopls, gum, jj, jq, K9s, kubectl, lazygit, nnn, ripgrep, sesh, tmux, and zoxide.

The `user` artifact also builds repository-defined configuration artifacts:

- Bat theme downloaded from the Tokyo Night upstream raw GitHub URL.
- Bat config generated by `BatConfig`.
- Claude Code settings generated by `ClaudeCode`.
- Codex config generated by `Codex`.
- OpenCode config generated by `Opencode`.
- Ghostty config generated by `GhosttyConfig`.
- K9s skin generated by `K9sSkin`.
- Neovim markdown ftplugin content generated by `FileCreate`.
- Claude status line script generated from `src/user/statusline.sh`.
- Claude `TeammateIdle` hook generated from `src/user/teammate-idle-hook.sh`.
- Claude agent definitions copied from `agents/claude-code`.
- Codex agent definitions copied from `agents/codex`.
- Claude skill definitions copied from `skills/claude-code`.
- Codex skill definitions copied from `skills/codex`.

The crate-level `get_output_path(namespace, digest)` helper constructs paths under `/var/lib/vorpal/store/artifact/output/{namespace}/{digest}`. `src/user.rs` uses that helper to derive generated artifact paths before creating host symlinks.

## Agent And Skill Assets

The repository contains parallel agent and skill source trees:

- `agents/claude-code/*.md`
- `agents/codex/*.toml`
- `skills/claude-code/*/SKILL.md`
- `skills/codex/*/SKILL.md`

`src/user.rs` copies the Claude and Codex source trees into Vorpal file-source artifacts and symlinks them into the corresponding host agent and skill locations. Codex agent roles are also declared inside the generated Codex TOML config with role descriptions, config file paths, and nickname candidates for team-lead, project-manager, staff-engineer, security-engineer, senior-engineer, sdet, and ux-designer.

The repository also contains `.claude/agent-memory` and `.claude/skills/evolve-*` files. Those are present in the workspace, but this architecture pass did not find references to them in `src/user.rs` or `Vorpal.toml` source includes.

## External Integrations

The build integrates with these external systems:

- Vorpal SDK and runtime for artifact definitions, artifact context, content-addressed outputs, environment definitions, and symlink publication.
- `vorpal-artifacts` from `https://github.com/ALT-F4-LLC/artifacts.vorpal.git` on branch `main`.
- Remote source downloads listed in `Vorpal.lock`, including GitHub release assets, `sdk.vorpal.build`, Kubernetes downloads, AWS CLI downloads, and other upstream package URLs.
- S3-backed Vorpal registry cache `altf4llc-vorpal-registry` in GitHub Actions.
- GitHub Actions workflow `.github/workflows/vorpal.yaml`, which runs a shell hook test on Ubuntu, builds `dev` on macOS, uploads `Vorpal.lock`, and builds `user` on macOS.
- AI tool integrations through generated Claude Code, Codex, and OpenCode configs, including model settings, sandbox/permission settings, telemetry endpoints, hooks, skills, and agent role mappings.

## Dependency Graph

The Rust package depends directly on:

- `anyhow` for error handling.
- `indoc` for multi-line formatted strings.
- `serde` and `serde_json` for structured config serialization.
- `tokio` for the async runtime used by the Vorpal binary.
- `toml` for Codex TOML serialization.
- `vorpal-sdk` for artifact APIs, environment types, and build context.
- `vorpal-artifacts` for reusable third-party CLI artifact definitions.

`cargo tree --depth 2` confirms `vorpal-artifacts` also depends on `anyhow`, `indoc`, `tokio`, and `vorpal-sdk`. No separate workspace members were observed in `Cargo.toml`.

## Generated Artifacts And Lockfiles

Generated runtime/build outputs are kept outside source control or in lockfiles:

- `target/` is ignored by `.gitignore`.
- `.docket/` is ignored by `.gitignore`.
- `Vorpal.lock` is checked in and records source URLs, digests, and platforms for remote inputs.
- GitHub Actions uploads `Vorpal.lock` from the `build-dev` job as an artifact.

The code treats Vorpal store output paths as content-addressed by digest and expects generated files and copied trees to be consumed through store paths rather than edited in place.

## Gaps & Risks

- `Vorpal.toml` includes only `src`, `Cargo.toml`, and `Cargo.lock`, while `src/user.rs` copies `agents/*` and `skills/*` through `FileSource`. It is not clear from repository-local evidence whether changes to `agents` or `skills` participate in Vorpal source invalidation without being listed in `Vorpal.toml`.
- `SYSTEMS` declares four target systems, but the observed lockfile entries are for `aarch64-darwin`, and the README states Apple Silicon macOS is primary. Cross-platform support beyond that platform is declared in code but not evidenced by the checked-in lock snapshot.
- `src/user.rs` is the central composition point for tool selection, generated config values, permissions, environment variables, copied source trees, and symlinks. That concentration makes architecture review of user-environment changes important because unrelated concerns can change in one file.
- Several generated configs contain operational, security, telemetry, and UX behavior in the same builder chains. Dedicated security, operations, and code-quality specs should own detailed review criteria for those values.
- The repository contains `.claude` local assets that are not wired into the observed Vorpal user artifact path. Their intended lifecycle is not documented in the architecture sources reviewed here.
- The symlink for `~/.vorpal/bin/vorpal` points at a hard-coded repository path under `$HOME/Development/repository/github.com/ALT-F4-LLC/vorpal.git/main/target/debug/vorpal`, which appears external to this repository and may make the user artifact dependent on a sibling checkout/build.
