# Recurring orchestration pitfalls — team-lead

- **Symptom:** teammates ack "duplicate dispatch" or hold for a GO that was already sent; team-lead receives its own task assignments back as teammate messages. **Root cause:** `TaskUpdate owner=<name>` emits a `task_assignment` notification that rides the same async channel as SendMessage briefs — echoes race/duplicate the authoritative brief (observed 3× in one session: tasks 1-3 echoed back to team-lead; planner ack'd a "duplicate"; advisor held for GO after processing the task_assignment before the GO landed). **Resolution:** treat `task_assignment` notifications as state-sync noise, never as instruction; the SendMessage brief is the sole authoritative dispatch; when a teammate blocks on a crossed message, answer its direct question ONCE (with any new evidence, e.g. termination timestamps) rather than re-sending the original brief.
