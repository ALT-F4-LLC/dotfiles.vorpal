- Historical-audit signal re-fires for an already-encoded lesson → cross-project pitfalls memory is never cleared, so the same fem-kubernetes render-mandate signal surfaced in 3 consecutive evolve-skills cycles (2026-06-04 add, 2026-06-08 no-op, 2026-06-09 no-op) against design-qa → before drafting any CHANGE from an audit signal, grep the target skill AND its changelog for the lesson first; a verified already-encoded signal is a no-op verdict, not a re-add.
- evolve-* historical audits report FP-dominated operator-correction counts (3 consecutive cycles, 2026-06-09 cycle 6/6 raw hits false) → the correction-phrase regex (`didn't work|actually|broken|...`) matches audit-report prose echoed inside teammate-message relays and command output replicated into user-role transcript turns, not operator typing → filter to operator-typed turns (exclude turns containing `<teammate-message`/`<command-name>`/`tool_result` markers) before counting; until the skill encodes this, treat raw correction counts as ~0 true positives.
- Bash snippets in skill bodies silently corrupted by argument substitution → Claude Code expands unescaped `$`+digit in SKILL.md content as 0-based positional args (doc-confirmed v2.1.163+: `$N` = `$ARGUMENTS[N]`, escape is `\$N`), so a shell-function recipe using `$1`/`$2`/`$3` gets its params replaced by stray invocation words → when auditing/authoring any skill containing bash recipes, grep `grep -nE '\$[0-9]'` and escape hits as `\$N`; substitution restores the literal `$N` before the agent reads the body (found in review-and-comment 2026-06-09, the only affected skill of 15 that cycle).
- Pre-flight hard-refusal gate ordered AFTER an interactive overwrite dialog → numbered procedure steps drift from the Failure Mode table's precedence qualifiers (e.g., prd's "file exists (and slug is not reserved)" implied reserved-check-first, but steps ran collision dialog at 4 and refusal at 5, exposing "Overwrite" on init-specs-owned reserved specs) → when reviewing any skill with both a refusal gate and a collision/overwrite dialog, verify the abort-class check precedes the interactive dialog in step order, not just in the table (found in prd 2026-06-09).
- Skill bodies instruct direct `Skill(vote, ...)` while the calling agent's spec bans it in team mode → report-emission skills' Save & Return / When-NOT-to-Use sections predate (or drifted from) the vote Delegation Protocol, so they tell the caller to nest a team (banned by every agent spec's CRITICAL banner; correct path is `docket vote create` + delegation_request to team-lead) → when reviewing any skill that mentions vote escalation, grep `Skill(vote` and require mode-split phrasing (standalone: direct; team: delegation); found in design-review:237/85, code-review:390/96, verify-ac:261/93 (2026-06-09 cycle — design-review fixed, siblings routed to Phase 2).
- Prior-cycle "verified accurate" verdicts on cross-file references treated as still valid → a changelog no-change entry records verification against a SNAPSHOT (referenced agent file, CLI version), and both drift — simplify-scout's 12-principle table was verified 2026-06-08 but agents/senior-engineer.md carried uncommitted evolve-agents modifications by 2026-06-09, and the 2026-05-29 changelog's "bundled /simplify renamed → /code-review in 2.1.147" claim was false against the live v2.1.170 skill list (bundled /simplify exists, applies fixes) → before reusing any prior-cycle verification, check `git status` for modifications to the referenced file and re-confirm CLI/feature claims against the current environment; re-verify rather than cite the old changelog.
- Prior no-rename verdicts treated as settled while the platform moves onto the name → "stability is load-bearing" was repeated across 16 code-review changelog entries, but a bundled skill later shipped under the same name (`/code-review`, observable as TWO same-name entries in the live session skill listing; bundled variant supports `--fix`, so mis-resolution breaks the leaf read-only contract) → the Rename dimension is a live check each cycle: inspect the CURRENT session skill listing for duplicate names instead of citing prior cycles' no-rename rationale; collision precedent `verify`→`verify-ac`, now code-review (rename recommended 2026-06-09).
- Changelog records an edit absent from the current skill file → out-of-cycle operator commits revert evolve-cycle edits without a changelog entry (2026-06-04 cycle trimmed Glob/Grep from all 4 doc-authoring skills' allowed-tools in 9540779; c10195b docs-path-taxonomy refactor re-added them family-wide, unrecorded), so later reviewers chase phantom changelog-vs-file drift (ux-spec's 2026-06-08 entry chased the same class; re-chased 2026-06-09) → before flagging changelog-vs-file drift as un-applied, run `git log -S'<old string>' --oneline -- <file>` to distinguish applied-then-reverted (legitimate, note as history gap) from never-applied (apply-verification defect); only the latter is a cycle defect.

## 2026-06-09 — NO-OP verdict citing a broken delivery channel as enforcement
symptom → prior evolve cycle (vote 2026-06-08) NO-OP'd the "silent-idle verdict capture" focus area by citing skill text "coordinator parses Agent()-returns" as already-enforced.
root cause → the cited channel does not exist for the skill's actual spawn shape: reviewers are spawned with `team_name` (teammates), and teammate plain final-turn text is NOT visible to the spawner — only SendMessage is. Reviewing the text against the complaint, not against envelope semantics, validated a dead channel.
resolution → when a skill/agent claims an output-retrieval channel, verify the channel against the spawn mode (fork subagent return vs teammate SendMessage-only) before NO-OP'ing; empirical pitfalls (reviewers idle, zero verdicts) outrank in-file claims.
- Skill-rename reference sweeps over-match without scope rules → a raw `grep code-review` hits historical changelog entries, append-only pitfalls memory, prose "code review", and bundled-skill mentions, none of which should be rewritten → scope rename manifests to LIVE definition files only (agents/*.md, skills/*/SKILL.md, .claude/skills/*/SKILL.md); changelogs get only their own file renamed + a new Rename entry; memory files are never edited; use token-boundary patterns (`Skill\(name[,)]`, `skills/name/`) so the new hyphen-extended name (`code-review-verdict`) is not re-matched (2026-06-09 cycle, precedent verify→verify-ac).
- Phase 0 docs-researcher asserts a false CURRENT-repo-state claim → 2026-06-09 evolve-agents briefing stated "our agents currently use none of the optional frontmatter fields" while all 7 agents had carried `effort` + `memory: project` since 2026-03-20 (researcher grounded feature EXISTENCE in fetched docs but never grepped local frontmatter for ADOPTION state) → treat researcher repo-state claims as unverified signals distinct from feature claims: before drafting any adoption CHANGE, `head -20 agents/*.md | grep` the actual frontmatter; a satisfied recommendation is a NO-OP and the false claim must be flagged in Coherence Issues so sibling reviewers acting on the same briefing don't propose re-adds.

## 2026-06-10 — Recorded model constraint scoped narrower than the platform reference
symptom → team-lead agent memory (project_effort_settings.md) records "Haiku 4.5 errors on xhigh/max effort", implying high/medium are accepted; the claude-api reference states the effort parameter errors on Haiku 4.5 at EVERY level — the memory under-states the constraint because only the failing level in use (xhigh) was ever observed.
root cause → empirical memory entries record the scope of the test that produced them, not the scope of the underlying constraint; later evaluations (model routing, "reevaluate effort/model" requests) treat the recorded scope as exhaustive.
resolution → before citing any memory-recorded model/API capability constraint in a routing or design verdict, cross-check it against the claude-api skill reference (or Models API capabilities tree) and report both scopes with the platform doc as controlling; mark the memory's narrower claim as an under-test, not a contradiction.

## 2026-06-10 — TDD line-budget feasibility asserted from estimate, not measurement
symptom → evolutionary-core TDD claimed "~18-22 reclaimable lines/file" to fit a new CANONICAL block under the 500-line gate (13/9 lines headroom); both secondary reviewers measured the absorbable sections at 13/16 physical lines and projected the phase net-ADDITIVE — the foundation phase would have breached the hard gate.
root cause → budget-feasibility math was composed from the design's intent ("the block absorbs that prose") without drafting the actual replacement text or running `sed -n`/`wc -l` on the target regions; same class as composing AC greps from design prose — a load-bearing quantitative claim whose target was never measured.
resolution → any TDD claim that an edit fits a line budget MUST ship the actual proposed text plus post-insertion arithmetic (current wc-l − measured absorbed + measured added ≤ budget) per file, executable-by-inspection; exploit physical-newline counting (one dense paragraph ≈ 1 line — the BANNER precedent) when a table/list would be net-additive; and when later phases add lines, attach a per-phase budget ledger with NAMED offsetting trims, gated on post-apply wc-l (fix shipped in the 2026-06-10 revision; ledger under-delivered at Phase C — headroom landed 4/2 — so the ledger's trim rows need the same measurement rigor as the foundation claim).

## 2026-06-10 — Changelog phase-attribution drifts from the TDD's phase plan
symptom → evolutionary-core cycle's three changelog entries each attributed the SAME shared CANONICAL:EVOLUTION-MODEL block to a DIFFERENT phase (evolve-agents "Phase A", evolve-skills "Phase B", evolve-coherence "Phase C"), but the governing TDD defines Phase A as the single foundation step that inserts the block byte-identically into all three carriers — the per-file changelog author labeled the bullet with whatever phase touched that file rather than the phase that owns the block.
root cause → when one logical change (a parity block) lands across N files via N separate changelog writers, each writer narrates from its own file's perspective and invents a local phase label; nothing cross-checks the label against the TDD's phase ownership.
resolution → when a change is a CANONICAL/parity block governed by a TDD with named phases, the changelog bullet must cite the phase that OWNS the block in the TDD (same label in all carriers), not the per-file edit order; in review, grep the phase labels across the family's changelogs and assert they agree for a shared-block bullet.
