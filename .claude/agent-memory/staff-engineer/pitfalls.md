## 2026-06-12 — Bash template carries shell state across tool calls (PID in a variable dies between calls)
Symptom: a review-fix replaced a working `pkill -f` with `PIDS[<name>]=$\!` set in the launch Bash call and `kill "${PIDS[<name>]}"` in later kill calls — both kills expand empty (error swallowed by `2>/dev/null`), so no run ever terminates and the downstream "after all terminate" gate blocks forever. Root cause: harness Bash calls do not share shell state (env vars/arrays die between calls); the template was reviewed for kill-mechanism correctness but not for VARIABLE LIFETIME across the tool-call boundary. Resolution: when reviewing any multi-step Bash template in a skill/agent doc, trace every variable from set-site to use-site and flag any pair living in different tool calls — cross-call state must persist to disk (pid-file, state file) or ride the orchestrator's context; also treat `2>/dev/null` on a command consuming such a variable as a masking smell.

## 2026-06-12 — Implementer addition beyond the TDD procedure ships an unverified platform claim
Symptom: the evolve-suite serial rewrite's crash-re-entry runbook instructed a NEW session to run the team-free guard (TeamDelete-by-name) against the dead session's team — a step the governing TDD's crash-re-entry procedure never prescribed; whether TeamDelete works from a session that never led that team is unverified platform behavior, yet it shipped as a confident procedure line. Root cause: review attention concentrates on TDD-conformance (does the implementation match the prescribed steps?), so implementer ADDITIONS — text with no TDD counterpart to diff against — bypass the executable-claim/evidence bar that TDD claims themselves had to pass; an addition reads as "extra thoroughness" and inherits unearned trust. Resolution: in code review of any skill/agent/runbook implementation, explicitly diff the artifact AGAINST the TDD in both directions — conformance (TDD steps present) AND additions (artifact steps absent from the TDD) — and hold each addition to the same evidence bar as a TDD claim (verify the platform behavior, or require self-healing phrasing like "attempt X; on failure fall through to <documented recovery path>"); an addition whose failure mode is NOT self-healing escalates from Concern to Blocker.
