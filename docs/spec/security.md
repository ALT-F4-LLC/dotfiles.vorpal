# Security Specification

## Security Context

This repository is a Vorpal-based Rust project that builds two artifact families: `dev`, a development toolchain, and `user`, a full user environment. The user environment installs CLI tools, generated configuration files, Claude Code and Codex agent definitions, skills, shell hooks, and host symlinks into user-owned paths.

Security-sensitive assets include local home-directory credentials, generated AI-tool permission policy, generated agent and skill prompts, S3-backed Vorpal registry credentials in CI, external artifact sources, and the host filesystem locations targeted by symlinks.

## Trust Boundaries

- Repository source enters the build through `Vorpal.toml`, which includes `src`, `Cargo.toml`, and `Cargo.lock`.
- Vorpal build steps write generated files to `$VORPAL_OUTPUT` and resolve final paths under `/var/lib/vorpal/store/artifact/output/{namespace}/{digest}`.
- Generated user configuration crosses from immutable store artifacts into mutable host configuration through symlinks under `$HOME/.claude`, `$HOME/.codex`, `$HOME/.agents/skills`, `$HOME/.config`, `$HOME/.vorpal/bin`, and macOS `Library/Application Support` paths.
- AI-tool runtime policy is generated from Rust builders, not hand-authored JSON/TOML output. Security review must inspect both the builder calls in `src/user.rs` and serialization structs under `src/user/`.
- CI crosses from GitHub-hosted runners into the S3-backed Vorpal registry through repository secrets and variables.
- External dependency and artifact trust enters through crates.io packages, a Git dependency on `ALT-F4-LLC/artifacts.vorpal.git`, the `ALT-F4-LLC/setup-vorpal-action`, and a downloaded Tokyo Night theme from `raw.githubusercontent.com`.

## Permissions And Sandboxing

Claude Code configuration generated by `src/user.rs` enables sandboxing, requires sandbox availability, sets permission default mode to `auto`, and disables bypass-permissions mode. It denies read, edit, and write access to common secret and host-sensitive paths including `.env`, `.env.*`, `~/.aws`, `~/.ssh`, `~/.gnupg`, `~/.kube`, `~/.netrc`, `~/.doppler`, `~/.talos`, `~/.claude.json`, `~/.codex`, `~/.gemini`, `~/.opencode`, `~/.vorpal`, `/Applications`, `/Library`, `/System`, `~/Desktop`, and `~/Downloads`.

Claude Code also allows many read, build, test, and package-manager shell patterns, allows `WebSearch`, and allows `WebFetch` to `api.github.com`, `crates.io`, and `github.com`. Destructive or publication-sensitive shell operations are treated differently: `chown`, `git commit`, `git push`, and `rm` are ask-mode; `git checkout` and `git reset` are denied.

The same Claude Code configuration leaves several important escape valves in place. Sandbox auto-allows bash when sandboxed, permits unsandboxed commands, and excludes `aws`, `docker`, `gh`, `git`, and `kubectl` from sandboxing. Those exclusions are practical for developer workflows but make the CLI credentials and local cluster contexts for those tools part of the trusted host boundary.

Codex configuration generated by `src/user.rs` sets `approval_policy = "on-request"`, `approvals_reviewer = "auto_review"`, `sandbox_mode = "workspace-write"`, and `default_permissions = ":workspace"`. It disables analytics and feedback, stores CLI authentication in the OS keyring, stores MCP OAuth credentials automatically, enables shell, hooks, memories, and multi-agent features, and leaves agent reasoning hidden from raw display.

Codex shell environment policy inherits all environment variables and excludes only `AWS_*`, `AZURE_*`, and `GCP_*`. This reduces cloud credential exposure but does not block other secret-bearing variables such as provider tokens, service-specific API keys, or custom credential names.

OpenCode configuration generated by `src/user.rs` disables autoupdate, asks for edits, allows reads, glob/list/LSP operations, allows WebFetch, and asks for bash by default while allowing a small set of read-only shell prefixes such as `cat`, `find`, `grep`, `head`, `ls`, `sort`, `tree`, and `wc`.

The checked-in `.claude/settings.local.json` grants additional local WebFetch domains for Claude Code, including Claude documentation and an internal metrics endpoint. This file is repository-controlled and affects local Claude Code behavior when present.

## Secrets And Environment Variables

No hard-coded secret values were observed in the inspected Rust builders, shell hooks, CI workflow, Renovate config, or local Claude settings. The repository does declare secret paths and secret environment variable patterns that must remain protected.

CI injects `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, and `AWS_DEFAULT_REGION` into `ALT-F4-LLC/setup-vorpal-action` for the S3-backed registry bucket `altf4llc-vorpal-registry`. These values are sourced from GitHub Actions secrets and variables, not committed source.

The user environment installs `awscli2`, `doppler`, `direnv`, `gh`, `kubectl`, and `k9s`, all of which can interact with external credentials or privileged remote systems. The generated configs deny common credential paths to Claude Code, but the installed tools themselves remain available to the user and, in some cases, are explicitly excluded from Claude Code sandboxing.

Claude Code generated environment sets telemetry-related OTEL endpoints and sets `CLAUDE_CODE_SUBPROCESS_ENV_SCRUB=0`. That setting should be treated as a deliberate secret-exposure risk until there is a documented reason and abuse-case coverage for subprocess environment propagation.

Codex generated environment excludes major cloud credential prefixes but otherwise inherits the shell environment. Any future addition of provider API keys, deployment credentials, or internal service tokens should either use the tool's credential store or extend the exclusion list.

## Remote Access And Network Egress

The README documents S3-backed remote caching for Vorpal artifacts through bucket `altf4llc-vorpal-registry`.

GitHub Actions runs on `ubuntu-latest` for hook tests and `macos-latest` for Vorpal builds. The build jobs use `ALT-F4-LLC/setup-vorpal-action@main`, then run `vorpal build 'dev'` or `vorpal build '${{ matrix.artifact }}'`.

Claude Code telemetry is configured to send logs and metrics to `https://loki.bulbasaur.altf4.domains/otlp/v1/logs` and `https://mimir.bulbasaur.altf4.domains/otlp/v1/metrics`. Codex OTel configuration points to the same log and metrics endpoints with `log_user_prompt = false`.

Claude Code sandbox network allow-list includes `crates.io`, `static.crates.io`, `github.com`, and `api.github.com`, and disables local network binding. Claude Code permission policy separately allows WebFetch for GitHub and crates.io and allows WebSearch.

OpenCode allows WebFetch in its generated permission block. No repository-local OpenCode network domain allow-list was observed.

The Bat theme is downloaded from a GitHub raw URL pinned to `folke/tokyonight.nvim` tag `v4.14.1`. Renovate tracks this URL through a custom regex manager.

## Generated Files And Host Installation

Generated file artifacts are created through `FileCreate`, which writes heredoc content into `$VORPAL_OUTPUT/{name}` and sets mode `0644` or `0755` when executable. The status line and teammate-idle hook are executable generated scripts.

`FileDownload` accepts a name and URL/path, wraps it as an `ArtifactSource`, and copies source contents to `$VORPAL_OUTPUT`. `FileSource` copies selected repository directories into store artifacts for agents and skills.

The user environment symlinks generated artifacts into host configuration paths. Security review for changes to symlink targets must consider both the source artifact and the destination path, especially for `$HOME/.claude/settings.json`, `$HOME/.codex/config.toml`, `$HOME/.claude/agents`, `$HOME/.claude/skills`, `$HOME/.agents/skills`, `$HOME/.config/opencode/opencode.json`, and `$HOME/.vorpal/bin/vorpal`.

The generated `$HOME/.vorpal/bin/vorpal` symlink points to a development checkout path under `$HOME/Development/repository/github.com/ALT-F4-LLC/vorpal.git/main/target/debug/vorpal`, not to a store artifact. That makes local checkout integrity part of the runtime trust model.

## Hooks And Local Scripts

The Claude Code status line script reads JSON from stdin, extracts fields with `jq`, runs `git -C "$PROJECT_DIR"` for repository status, and writes a cache file under `/tmp/claude-statusline-git-cache-{cksum}`. It does not perform network access.

The teammate-idle hook reads JSON from stdin, extracts `agent_type` with `jq`, and emits a system message as JSON. It fails closed to `{}` when input parsing or `jq` availability fails and does not perform network or filesystem writes beyond process output.

Hook configuration is enabled for both Claude Code and Codex. Any future hook that runs shell commands should be reviewed as a privilege-boundary change because hooks run automatically in response to tool events or session state.

## Supply Chain

Direct Rust dependencies in `Cargo.toml` are `anyhow`, `indoc`, `serde`, `serde_json`, `tokio`, `toml`, `vorpal-artifacts`, and `vorpal-sdk`. `vorpal-artifacts` is declared as a Git dependency on branch `main`; `Cargo.lock` pins the resolved commit. `vorpal-sdk` is sourced from crates.io.

`Cargo.lock` is present and records checksums for registry dependencies. Renovate automerges minor and patch updates for stable Cargo crates, groups `serde` and `serde_json`, requires manual review for major Cargo updates, and tracks the Tokyo Night theme version.

GitHub Actions uses mutable action references rather than commit SHAs for `actions/checkout@v6`, `actions/upload-artifact@v7`, and especially `ALT-F4-LLC/setup-vorpal-action@main`. The `@main` reference is the highest supply-chain risk because it can change without a version tag update in this repository.

## Security Review Expectations

- Treat changes to `src/user.rs`, `src/user/claude_code.rs`, `src/user/codex.rs`, `src/user/opencode.rs`, `.github/workflows/vorpal.yaml`, `Cargo.toml`, `Cargo.lock`, `renovate.json`, `agents/**`, `skills/**`, and hook scripts as security-relevant when they alter permissions, sandboxing, credential paths, network access, generated symlinks, or dependency sources.
- Rebuild or inspect generated config after changing builders so reviewers can verify the emitted JSON/TOML policy, not just builder intent.
- Verify that new AI-tool permissions preserve deny-lists for credential paths and require explicit approval for destructive, publishing, and credential-bearing commands.
- Route auth, secrets, sandbox, permission, telemetry, CI credential, and supply-chain changes to the security-engineer role defined under `agents/codex/security-engineer.toml`.

## Gaps & Risks

- There is no automated test observed that snapshots or validates the generated Claude Code, Codex, or OpenCode security policy.
- `CLAUDE_CODE_SUBPROCESS_ENV_SCRUB=0` and Codex `inherit = "all"` leave non-cloud-prefixed environment secrets exposed unless each tool applies additional runtime controls outside this repository.
- Claude Code allows unsandboxed commands and excludes `aws`, `docker`, `gh`, `git`, and `kubectl` from sandboxing; those tools can access credentials, remote systems, or mutable host state.
- `ALT-F4-LLC/setup-vorpal-action@main` is not pinned to an immutable commit SHA.
- Renovate automerge for stable minor and patch Cargo updates can merge security-sensitive dependency changes without human review when the package version is `>= 1.0.0`.
- The generated Vorpal binary symlink targets a local development checkout rather than a content-addressed store artifact.
- OpenCode allows WebFetch without a repository-observed domain allow-list.
- The repository does not document a formal secret-scanning, dependency-audit, or generated-policy diff requirement in CI.
- No existing security spec was present to define accepted residual risk for telemetry endpoints, AI-tool memory/history persistence, MCP OAuth credential storage, or sandbox escape valves.
