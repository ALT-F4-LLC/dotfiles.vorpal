# Security

## Overview

dotfiles.vorpal is a declarative dotfiles manager that produces reproducible, content-addressed
artifacts via the Vorpal build system. From a security perspective, the project operates across
three domains: (1) a Rust build-time program that generates configuration files and fetches
external binaries, (2) a CI pipeline that handles AWS credentials for remote artifact caching,
and (3) generated AI agent configurations (Claude Code, OpenCode) that enforce filesystem and
command sandboxing on the developer workstation.

The project does not run a network service, does not process untrusted user input at runtime,
and does not store application-level secrets in its source tree. Its primary security concerns
are supply chain integrity, CI secret handling, and the AI agent permission boundaries it
configures.

## Trust Boundaries

### Boundary 1: Source Code to Build Artifacts

The Rust source code (`src/`) defines the entire environment declaratively. Vorpal's
content-addressed store (`/var/lib/vorpal/store/`) means artifacts are identified by their
content hash, providing tamper detection at the artifact level. The `Vorpal.lock` file pins
every external source with a SHA-256 digest, ensuring reproducible builds and preventing silent
upstream changes.

**Trust assumption:** The Vorpal runtime itself is trusted. The project depends on Vorpal to
enforce content addressing and to correctly execute build steps in isolation.

### Boundary 2: External Source Downloads

All CLI tools and themes are fetched from external URLs during the Vorpal build. Sources include
GitHub Releases, official project mirrors (rust-lang.org, go.dev, kernel.org), and AWS
(awscli2). Each download is pinned to a specific version and verified against a SHA-256 digest
in `Vorpal.lock`.

**Trust assumption:** The download sources are authentic at the time the digest is recorded.
There is no GPG signature verification or secondary attestation beyond the content hash.

### Boundary 3: CI/CD Pipeline to AWS

GitHub Actions workflows (`.github/workflows/vorpal.yaml`) use `AWS_ACCESS_KEY_ID` and
`AWS_SECRET_ACCESS_KEY` from GitHub Secrets to access the S3 artifact registry
(`altf4llc-vorpal-registry`). The `AWS_DEFAULT_REGION` is stored as a GitHub variable (not a
secret).

**Trust assumption:** GitHub Secrets are not exposed in logs or to pull request workflows from
forks (standard GitHub Actions behavior).

### Boundary 4: Generated Configs to Host Filesystem

The build output creates symlinks from the Vorpal store into the user's home directory
(`~/.claude/`, `~/.config/`, `~/Library/Application Support/`). These symlinks point into the
content-addressed store, so the content is immutable once built. However, the symlink targets
themselves are in user-writable locations.

**Trust assumption:** The user's home directory has standard macOS permissions. Symlink creation
is the only filesystem mutation the build performs outside `/var/lib/vorpal/store/`.

### Boundary 5: AI Agent Sandbox

The Claude Code configuration generated by `src/user/claude_code.rs` and deployed to
`~/.claude/settings.json` defines a three-tier permission model (allow/ask/deny) and a
filesystem sandbox. This is the most security-critical output of the project, as it governs
what an AI agent can do on the developer workstation.

## Secret Management

### Secrets in the Codebase

**No application secrets exist in the source tree.** The repository is clean:

- `.gitignore` excludes `/target` and `/.docket` -- no build artifacts or task state in git
- No `.env` files exist in the repository
- `.envrc` is present but access is restricted (it is used by direnv for local environment setup)
- The `api_key_helper` field in the Claude Code config struct exists but is not set to any value
  in the current build configuration (it is `None`)

### Secrets in CI

Two secrets are used in GitHub Actions:

| Secret | Scope | Purpose |
|---|---|---|
| `AWS_ACCESS_KEY_ID` | GitHub Secrets | S3 registry authentication |
| `AWS_SECRET_ACCESS_KEY` | GitHub Secrets | S3 registry authentication |

Both are accessed via `${{ secrets.* }}` syntax, which prevents exposure in logs. The CI workflow
uses a third-party action (`ALT-F4-LLC/setup-vorpal-action@main`) pinned to the `main` branch
rather than a specific commit SHA.

### Secrets at Runtime

The generated user environment references external services that require credentials managed
outside this project:

- **AWS CLI** (`awscli2`) -- credentials managed via `~/.aws/` (denied from AI agent read access)
- **Doppler** -- secrets management tool; its own credentials are in `~/.doppler/` (denied from
  AI agent read/write access)
- **GitHub CLI** (`gh`) -- authentication managed via `gh auth login`; tokens stored outside the
  project
- **Kubernetes** (`kubectl`, `k9s`) -- kubeconfig at `~/.kube/` (denied from AI agent
  read/write access)

### Telemetry Endpoints

The Claude Code configuration sends telemetry data to external observability endpoints:

- Logs: `https://loki.bulbasaur.altf4.domains/otlp/v1/logs` (OTLP over HTTP/protobuf)
- Metrics: `https://mimir.bulbasaur.altf4.domains/otlp/v1/metrics` (OTLP over HTTP/protobuf)

These endpoints use HTTPS. No authentication tokens for these endpoints are present in the
source code -- they are presumably configured externally or the endpoints accept unauthenticated
writes.

## AI Agent Permission Model

The Claude Code configuration in `src/user.rs` (lines 86-257) defines the most detailed
security policy in the project. It implements defense-in-depth through three mechanisms:

### 1. Command Permissions (Allow / Ask / Deny)

**Allowed commands** (auto-approved, no user prompt):

- Read-only operations: `git status`, `git log`, `git diff`, `git show`, `ls`, `cat`, `head`,
  `tail`, `find`, `grep`, `rg`, `tree`, `wc`, `sort`, `jq`
- Build tools: `cargo build/check/clippy/fmt/test/run/tree/update/search/outdated`,
  `go build/test/vet/doc/list`, `bun run/test`, `npm run build/lint/test`, `yarn build/lint/test`
- Infrastructure read: `docker images/logs/ps`, `vorpal build/inspect`, `gh pr diff/list/view`
- Other: `docket`, `cue`, `make`, `staticcheck`, `gofmt`, `npx tsc`, `tar`, `xargs`, `chmod`,
  `test`

**Ask commands** (require explicit user approval each time):

- `git commit` -- prevents unauthorized commits
- `git push` -- prevents unauthorized pushes
- `rm` -- prevents accidental file deletion
- `chown` -- prevents ownership changes

**Denied commands** (blocked entirely):

- `git checkout` -- prevents branch switching that could discard work
- `git reset` -- prevents destructive history rewriting

### 2. Filesystem Permissions (Read / Write / Edit)

Sensitive directories are denied for all three operations (Read, Write, Edit):

| Protected Path | Rationale |
|---|---|
| `~/.aws/**` | AWS credentials (Read only -- Write/Edit not explicitly denied) |
| `~/.ssh/**` | SSH keys |
| `~/.gnupg/**` | GPG keys |
| `~/.kube/**` | Kubernetes credentials |
| `~/.doppler/**` | Doppler secrets |
| `~/.netrc` | Network credentials |
| `~/.talos/**` | Talos cluster config |
| `~/.claude/**` | Claude Code's own config (prevents self-modification) |
| `~/.claude.json` | Claude Code user config |
| `~/.codex/**` | Codex config |
| `~/.gemini/**` | Gemini config |
| `~/.opencode/**` | OpenCode config |
| `~/.vorpal/**` | Vorpal config |
| `/Applications/**` | System applications |
| `/Library/**` | System library |
| `/System/**` | System files |
| `~/Desktop/**` | User desktop |
| `~/Downloads/**` | User downloads |
| `~/Library/**` | User library |
| `.env`, `.env.*`, `.envrc` | Environment files (Read denied) |

**Notable design decision:** The agent cannot read `.env`, `.env.*`, or `.envrc` files, which
prevents accidental exposure of environment-level secrets in AI agent context.

### 3. Sandbox Configuration

- **Sandbox enabled:** `true`
- **Auto-allow bash if sandboxed:** `true` -- bash commands run in the sandbox without prompting
- **Allow unsandboxed commands:** `true` -- specific commands can opt out of sandboxing
- **Sandbox-excluded commands:** `aws`, `docker`, `gh`, `git` -- these run unsandboxed because
  they need access to credentials and network resources the sandbox would block
- **Network local binding:** `false` -- prevents the agent from binding local network ports
- **Permission bypass disabled:** `"disable"` -- prevents the agent from escalating its own
  permissions
- **Default mode:** `"acceptEdits"` -- agent can propose edits that the user reviews

### OpenCode Permission Model

The OpenCode configuration (`src/user.rs`, lines 263-291) uses a simpler permission model:

- Default bash command permission: `Ask` (requires approval)
- Read-only commands allowed without approval: `cat`, `echo`, `file`, `find`, `git branch`,
  `git log`, `grep`, `head`, `ls`, `sort`, `test`, `tree`, `wc`
- File edit: `Ask`
- File read, glob, list, LSP: `Allow`
- Web fetch: `Allow`

## Supply Chain Security

### Rust Dependencies

The project has 268 transitive Cargo dependencies (per `Cargo.lock`). Key security-relevant
dependencies:

| Dependency | Version | Purpose |
|---|---|---|
| `tonic` | 0.14.5 | gRPC client (communicates with Vorpal daemon) |
| `rustls` | 0.23.37 | TLS implementation (used by tonic for HTTPS) |
| `ring` | 0.17.14 | Cryptographic primitives (used by rustls) |
| `hyper` | 1.x | HTTP implementation (transport layer for tonic) |
| `tokio` | 1.x | Async runtime |
| `serde` / `serde_json` | 1.x | JSON serialization for config generation |

TLS is handled by `rustls` with `ring` as the crypto backend, which is a memory-safe alternative
to OpenSSL. Native certificate verification is provided via `rustls-native-certs` (uses the
system certificate store) and `security-framework` on macOS.

### Dependency Management

Renovate is configured (`renovate.json`) with the following security-relevant policies:

- **Auto-merge:** Minor and patch updates for stable crates (version >= 1.0) are auto-merged
- **Manual review:** Major version updates require manual review
- **Pre-release crates:** The `vorpal-sdk` dependency is at `0.1.0-alpha.0`, which is excluded
  from auto-merge by the `matchCurrentVersion: "!/^0/"` rule

### External Binary Sources

All CLI tools in `Vorpal.lock` are fetched from official sources over HTTPS:

- GitHub Releases (bat, direnv, doppler, fd, gh, jj, jq, k9s, lazygit, nnn, ripgrep, tmux)
- Official project mirrors (git from kernel.org, go from go.dev, rust toolchain from
  rust-lang.org, kubectl from dl.k8s.io)
- AWS (awscli2 from awscli.amazonaws.com)

Each binary is pinned to a specific version and SHA-256 digest. **There is no GPG signature
verification** -- integrity relies solely on the content hash matching what was recorded when
the lockfile was last updated.

### Third-Party GitHub Actions

The CI pipeline uses:

- `actions/checkout@v6` -- official GitHub action, pinned to major version tag
- `actions/upload-artifact@v6` -- official GitHub action, pinned to major version tag
- `ALT-F4-LLC/setup-vorpal-action@main` -- organization-owned action, pinned to branch (not
  commit SHA)

**Gap:** Pinning to `@main` for `setup-vorpal-action` means a compromised main branch of that
repository could inject malicious code into the CI pipeline. Pinning to a specific commit SHA
would be more resilient.

## Build-Time Security Properties

### Content Addressing

Vorpal's content-addressed store (`/var/lib/vorpal/store/artifact/output/{namespace}/{digest}`)
ensures that:

- Artifacts are immutable once stored
- The same inputs always produce the same output path
- Tampered artifacts would have a different digest and would not match references

### Shell Script Generation

The `FileCreate` struct (`src/file.rs`) generates artifacts by writing content into a bash
script that uses a heredoc (`cat << 'EOF'`). The single-quoted `EOF` delimiter prevents
variable expansion within the heredoc, which is the correct approach for injecting arbitrary
content safely. File permissions are set explicitly (644 for regular files, 755 for executables).

### No Runtime Network Access

The built binary (`vorpal`) is a build-time tool only. It does not run as a daemon or service
on the user's machine. Network access occurs only during the Vorpal build process (to fetch
sources and communicate with the Vorpal daemon via gRPC/tonic).

## Gaps and Recommendations

### Current Gaps

1. **No GPG verification of external binaries.** Supply chain integrity relies entirely on
   SHA-256 content hashes recorded in `Vorpal.lock`. If a source was compromised at the time
   the hash was recorded, the compromised binary would be permanently pinned.

2. **CI action pinned to branch.** `ALT-F4-LLC/setup-vorpal-action@main` should be pinned to
   a specific commit SHA to prevent supply chain attacks via the action repository.

3. **No `cargo-audit` or `cargo-deny` in CI.** Known vulnerability scanning for Rust
   dependencies is not part of the CI pipeline.

4. **Telemetry endpoints lack visible authentication.** The OTEL endpoints for Loki and Mimir
   have no authentication tokens configured in the source code. If these endpoints accept
   unauthenticated writes, they could be targets for log/metric injection.

5. **AWS credentials use long-lived keys.** The CI pipeline uses `AWS_ACCESS_KEY_ID` /
   `AWS_SECRET_ACCESS_KEY` rather than OIDC-based federation (e.g., GitHub Actions OIDC with
   AWS IAM roles), which would eliminate long-lived credentials.

6. **`~/.aws/**` is only Read-denied for AI agents.** Write and Edit deny rules for `~/.aws/**`
   are not present in the Claude Code permission configuration, though the broader sandbox may
   provide coverage.

### What Is Done Well

1. **Comprehensive AI agent sandboxing.** The Claude Code permission model is thorough --
   sensitive directories are denied, destructive git commands are blocked, commits and pushes
   require approval, and permission bypass is disabled.

2. **Environment file protection.** `.env`, `.env.*`, and `.envrc` are explicitly denied from
   AI agent read access, preventing accidental secret exposure.

3. **Content-addressed artifacts.** Vorpal's store model provides tamper detection and
   reproducibility by design.

4. **Rust memory safety.** Using `rustls`/`ring` instead of OpenSSL for TLS eliminates a
   class of memory safety vulnerabilities. The codebase uses no `unsafe` blocks.

5. **Minimal attack surface.** The project produces static configuration files -- it does not
   run services, accept network connections, or process untrusted input at runtime.

6. **Renovate for dependency updates.** Automated dependency updates with manual review for
   major versions reduces the window for known vulnerability exploitation.
