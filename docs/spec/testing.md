# Testing Specification

## Current State

**This project has no tests.** There are zero `#[test]` functions, zero `#[cfg(test)]` modules,
no `tests/` directory, no test utilities, no fixtures, no mocking infrastructure, and no coverage
tooling configured anywhere in the codebase.

The CI pipeline (`.github/workflows/vorpal.yaml`) runs only `vorpal build` commands -- it does
not execute `cargo test`, `cargo clippy`, or `cargo fmt --check`. There is no test step in CI.

## What Exists

### Build-Time Verification Only

The only verification that currently exists is implicit: the project must compile successfully
via `vorpal build`. This catches type errors, missing imports, and syntax issues but does not
verify runtime behavior, configuration correctness, or edge cases.

### Rust Toolchain

The project uses Rust 2021 edition with the following test-relevant dependencies:

- **`serde` + `serde_json`**: Used for serialization of configuration structs (e.g.,
  `ClaudeCode`, `Opencode`). These are prime candidates for unit testing -- verifying that
  builder patterns produce correct JSON output.
- **`indoc`**: Used for multi-line string formatting in config generation (`GhosttyConfig`,
  `FileCreate`). Template output could be tested.
- **`anyhow`**: Error handling via `Result<T>`. Error paths are untested.
- **`tokio`**: Async runtime. All `build()` methods are async and interact with `ConfigContext`
  from `vorpal-sdk`, making them harder to unit test without mocking the SDK.

### Claude Code Permissions for Testing

The Claude Code configuration generated by this project includes `Bash(cargo test:*)` in its
allow list, indicating the project owner expects `cargo test` to be used in development
workflows -- even though no tests exist yet.

## What Should Be Tested

Given the nature of this project (a declarative configuration generator), the highest-value
tests would be:

### Unit Tests (Highest Priority)

1. **Serialization output of configuration structs**: The `ClaudeCode`, `Opencode`, `K9sSkin`,
   `BatConfig`, and `GhosttyConfig` structs all use builder patterns to construct configuration
   and serialize to JSON or plain text. Unit tests should verify that builder method chains
   produce the expected serialized output. These tests are pure logic and do not require the
   Vorpal SDK context.

2. **`FileCreate` content generation**: Verify that the shell script templates produced by
   `FileCreate::build()` contain correct content and permissions. The template interpolation
   (using `indoc::formatdoc!`) could produce malformed output if content contains special
   characters.

3. **`get_output_path` utility**: Simple pure function that formats a path string. Trivial to
   test, good for establishing the testing pattern.

### Integration Tests (Medium Priority)

4. **Builder + serialization round-trips**: For `ClaudeCode` specifically, verify that the
   generated JSON is valid and parseable, and that `serde` rename attributes (`camelCase`) and
   skip attributes produce the expected shape.

5. **`UserEnvironment` artifact wiring**: Verify that all artifacts are correctly wired into the
   environment (symlink paths are valid, environment variables are correctly formatted). This
   would require mocking or stubbing the `ConfigContext` from `vorpal-sdk`.

### End-to-End Tests (Lower Priority)

6. **Full `vorpal build` execution**: Already covered implicitly by CI, but not as a structured
   test. The current CI approach (building `dev` and `user` artifacts) is a reasonable e2e
   validation for a configuration project.

## Gaps and Risks

### Critical Gaps

- **No regression protection**: Changes to builder methods or serialization logic have no
  automated checks. A breaking change to `serde` attributes or field names would only be caught
  by manual inspection or downstream failures.

- **No CI linting or formatting checks**: `cargo clippy` and `cargo fmt --check` are not run
  in CI. Code quality is enforced only by convention.

- **No coverage tooling**: No `cargo-tarpaulin`, `cargo-llvm-cov`, or equivalent is configured.
  There is no coverage baseline or target.

### Constraints on Testability

- **Vorpal SDK dependency**: Most `build()` methods take `&mut ConfigContext` from `vorpal-sdk`,
  which creates a tight coupling to the Vorpal build system. Unit testing pure serialization
  logic (e.g., `serde_json::to_string_pretty(&self)` in `ClaudeCode::build`) is straightforward,
  but testing the full build pipeline requires either mocking the SDK or running a real Vorpal
  context.

- **External artifact dependencies**: `vorpal-artifacts` provides pre-built tool artifacts
  (awscli2, bat, direnv, etc.). These cannot be tested in isolation without the Vorpal registry.

- **Async everywhere**: All `build()` methods are async, requiring `tokio::test` for any test
  that exercises them.

## Recommendations for Future Testing

1. **Start with serialization tests**: Extract the JSON/text generation logic from `ClaudeCode`,
   `GhosttyConfig`, `BatConfig`, `K9sSkin`, and `Opencode` into testable units that don't
   require `ConfigContext`. Test the serialized output against expected snapshots.

2. **Add CI quality gates**: Add `cargo test`, `cargo clippy -- -D warnings`, and
   `cargo fmt --check` steps to `.github/workflows/vorpal.yaml`.

3. **Consider snapshot testing**: A crate like `insta` would be well-suited for this project,
   since the primary output is generated configuration files. Snapshot tests would catch
   unintended changes to config output.

4. **Coverage tooling**: `cargo-llvm-cov` is the modern standard for Rust coverage. Given the
   current state (0 tests), any coverage target would be aspirational -- start by establishing
   the tooling, not a percentage target.
